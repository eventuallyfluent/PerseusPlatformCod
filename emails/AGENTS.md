# Email Template Instructions

## Purpose

Owns transactional email templates for purchase confirmation, onboarding, and future platform emails.

## Ownership

- Email presentation templates live in this folder.
- Sending logic lives in `lib/email`.

## Local Contracts

- Email copy should be customer-facing and avoid admin/internal mechanics.
- Do not include secrets, raw provider payloads, or sensitive payment details in templates.
- Preserve clear post-purchase or onboarding next steps.
- Withdrawal acknowledgements must include the order identifier and exact submission timestamp and must not claim a refund completed before confirmation.

## Work Guidance

- Keep templates compatible with email-client rendering constraints.
- Reuse existing brand tone and plain fallback-friendly structure.
- Keep route URLs generated by calling code or shared helpers when possible.

## Verification

- Run `npm run lint` after template TypeScript/React changes.
- Verify any changed links match current route contracts.
